## Entry Point Chain

```
KronosCommandLineProcessor (@AutoService(CommandLineProcessor))
  Plugin ID: "kronos-compiler-plugin"
  CLI options: dump-ir (bool), dump-ir-path (string), dump-ir-mode (string), dump-ir-files (string), debug (bool)
    ↓
KronosCompilerPluginRegistrar (@AutoService(CompilerPluginRegistrar), supportsK2 = true)
  Reads CompilerConfiguration keys → creates KronosIrGenerationExtension
    ↓
KronosIrGenerationExtension.generate(moduleFragment, pluginContext)
  1. Creates KronosParserTransformer, runs moduleFragment.transform(transformer, null)
  2. Post-processes initCallSiteLambdas (for cross-module @KronosInit)
  3. Optionally dumps IR to files for debugging
```

## KronosParserTransformer

Single `IrElementTransformerVoidWithContext` traversing the entire module. Five responsibilities:

### Task 1: Collect KPojo Classes
Sources: `visitClassNew()` (superTypes check), `visitClassReference()` (::class), `visitConstructorCall()`, `visitCall()` (type arguments). All discovered KPojo classes are stored for later use by `KClassMapGenerator`.

### Task 2: Transform KPojo Classes
In `visitClassNew()`, when a class implements `KPojo`:
```kotlin
declaration.transform(KronosIrClassTransformer(pluginContext, declaration, errorReporter), null)
```

### Task 3: Dispatch DSL Lambda Transformers
In `visitFunctionNew()`, checks the extension receiver FqName:
```kotlin
when (extensionReceiverFqName) {
    kTableForSelectSymbol    → SelectTransformer
    kTableForSetSymbol       → SetTransformer
    kTableForConditionSymbol → ConditionTransformer
    kTableForSortSymbol      → SortTransformer
    kTableForReferenceSymbol → ReferenceTransformer
}
```
Each transformer is applied via `transformWith()` which wraps the function body in a new `irBlockBody`.

### Task 4: Handle @KronosInit
Functions annotated with `@KronosInit` → `KClassMapGenerator.generate()` injects a `kClassCreator` map (KClass → constructor lambda) for all collected KPojo classes. Call-site lambdas are collected into `initCallSiteLambdas` for post-processing.

### Task 5: Fix Typed Query Parameters
In `visitCall()`, after standard traversal, `TypeParameterFixer.shouldFix()` checks if the call is `queryList`/`queryOne`/`queryOneOrNull` on `KronosQueryTask`, `SelectClause`, `SelectFrom*`, or `SqlHandler`. If so, injects two compile-time arguments:
- `isKPojo: Boolean` — whether `T` implements `KPojo`
- `superTypes: List<String>` — full type hierarchy as FQN strings

## KPojo Class Augmentation

`KronosIrClassTransformer` visits each function/property in a KPojo class. For fake-override members from the `KPojo` interface, it replaces them with real implementations generated by `KronosClassBodyGenerator`.

### Generated Method Bodies

| Method | What It Generates |
|--------|------------------|
| `toDataMap()` | `mutableMapOf("col1" to this.prop1, "col2" to this.prop2, ...)` — respects `@Ignore` |
| `fromMapData(map)` | Creates new instance, sets each property from map via constructor + copy |
| `safeFromMapData(map)` | Same but uses `getSafeValue()` for type-safe conversion |
| `kronosColumns()` | `mutableListOf(Field("col", "prop", type, primaryKey, ...), ...)` for all properties |
| `get(name)` | `when(name) { "prop1" -> this.prop1; "prop2" -> this.prop2; ... }` |
| `set(name, value)` | `when(name) { "prop1" -> this.prop1 = value as Type; ... }` |
| `__tableName` | From `@Table("name")` annotation or `tableNamingStrategy.k2db(className)` |
| `__tableComment` | Extracted from KDoc comment on the class |
| `kronosTableIndex()` | Reads `@TableIndex` annotations → `mutableListOf(KTableIndex(...))` |
| `kronosPrimaryKey()` | Reads `@PrimaryKey` → `KronosCommonStrategy(enabled, field)` |
| `kronosCreateTime()` | Reads `@CreateTime` → `KronosCommonStrategy(enabled, field)` |
| `kronosUpdateTime()` | Reads `@UpdateTime` → `KronosCommonStrategy(enabled, field)` |
| `kronosLogicDelete()` | Reads `@LogicDelete` → `KronosCommonStrategy(enabled, field)` |
| `kronosOptimisticLock()` | Reads `@Version` → `KronosCommonStrategy(enabled, field)` |
| `kClass()` | Returns `this::class` (the actual KClass, not `KPojo::class`) |

### How Annotations Are Read
`AnnotationUtils.kt` provides:
- `findAnnotation(fqName)` — finds annotation on class/property
- `getAnnotationStringValue(annotation, index)` — reads string argument
- `getColumnName(property)` — `@Column("name")` or naming strategy
- `getTableName(classDeclaration)` — `@Table("name")` or naming strategy
- `collectAnnotations(property)` — gathers all Kronos annotations into a structured result
- `shouldIgnore(property)` — checks `@Ignore` / `@NotColumn`

### How Types Are Mapped
`TypeUtils.mapTypeToKColumnType()` maps Kotlin types to `KColumnType` enum:
- `Boolean` → BIT, `Byte/Short` → TINYINT, `Int` → INT, `Long` → BIGINT
- `Float` → FLOAT, `Double` → DOUBLE, `BigDecimal` → DECIMAL
- `String` → VARCHAR, `Char` → CHAR
- `LocalDate/java.sql.Date` → DATE, `LocalTime` → TIME
- `LocalDateTime/Date/java.sql.Timestamp` → DATETIME
- `ByteArray` → BLOB, `UUID` → CHAR(36)
- Serializable types → SERIALIZABLE

## DSL Lambda Transformers

Each transformer handles a specific `KTableFor*<T>` lambda. The compiler plugin rewrites the lambda body from user-friendly Kotlin expressions into Kronos IR (Field lists or Criteria trees).

### SelectTransformer
Handles `KTableForSelect<T>` lambdas (used in `select { }` blocks).
- On `visitReturn`, analyzes the return expression via `FieldAnalysis.analyzeAndBuildFieldList()`
- Converts `it.name + it.age` into `listOf(Field("name"), Field("age"))`
- Handles aliasing: `it.name as_ "n"` → `Field("name", alias="n")`
- Handles functions: `f.count(it.id)` → `FunctionField("count", [Field("id")])`

### SetTransformer
Handles `KTableForSet<T>` lambdas (used in `set { }` blocks for updates).
- Converts `it.name = "new"` into field-value pairs
- Handles `it.name = it.name + 1` for increment operations

### ConditionTransformer
Handles `KTableForCondition<T>` lambdas (used in `where { }`, `having { }`, `on { }` blocks).
- On `visitReturn`, delegates to `ConditionAnalysis.analyzeAndBuildCriteria()`
- Converts Kotlin boolean expressions into `Criteria` tree

### SortTransformer
Handles `KTableForSort<T>` lambdas (used in `orderBy { }` blocks).
- Converts `it.age.desc() + it.name.asc()` into sorted field list

### ReferenceTransformer
Handles `KTableForReference<T>` lambdas (used in `cascade { }` blocks).
- Converts field references into cascade field selections

## FieldAnalysis

`core/FieldAnalysis.kt` (~860 lines). Parses field DSL expressions into `Field` lists.

Key method: `analyzeAndBuildFieldList(expression, irBuilder)` → `List<IrExpression>` representing Field constructors.

Handles:
- **Property access**: `it.name` → `Field("name_col", "name", VARCHAR, ...)`
- **Plus operator**: `it.name + it.age` → two fields concatenated
- **Unary plus**: `+it.name` → single field
- **Aliasing**: `it.name as_ "alias"` → field with alias set
- **Function calls**: `f.count(it.id)` → `FunctionField("count", [(Field("id"), null)])`
- **Minus operator**: `it - it.id` → all fields except id (exclusion)
- **All fields**: `it.eq` → all columns of the entity

Each field is constructed as an `irCallConstructor` for the `Field` class with all metadata (columnName, name, type, primaryKey, dateFormat, tableName, cascade info, ignore actions, isColumn, length, scale, defaultValue, nullable, serializable, kDoc).

## ConditionAnalysis

`core/ConditionAnalysis.kt` (~900 lines). Parses condition DSL expressions into `Criteria` trees.

Key method: `analyzeAndBuildCriteria(expression, irBuilder)` → `IrExpression` representing a Criteria constructor.

Handles:
- **Equality**: `it.name == "test"` → `Criteria(field, Equal, value="test")`
- **Comparison**: `it.age > 18` → `Criteria(field, GreaterThan, value=18)`
- **In**: `it.id in listOf(1,2,3)` → `Criteria(field, In, value=[1,2,3])`
- **Like**: `it.name.like("%test%")` → `Criteria(field, Like, value="%test%")`
- **Between**: `it.age.between(18, 30)` → `Criteria(field, Between, value=[18,30])`
- **IsNull**: `it.name.isNull` → `Criteria(field, IsNull)`
- **Boolean logic**: `cond1 && cond2` → `Criteria(type=And, children=[cond1, cond2])`
- **Negation**: `!cond` → `Criteria(..., not=true)`
- **Property getters**: `it.name.eq` → equality check using the entity's current value
- **Raw SQL**: `"sql".asSql()` → `Criteria(type=Sql, value="sql")`
- **No-value strategy**: `.ifNoValue(ignore)` → sets `noValueStrategyType`
- **Conditional**: `.takeIf(boolean)` → conditionally includes criteria

The analysis recursively walks the IR expression tree, matching on `IrCall` function names and receiver types.

## TypeParameterFixer

`transformers/TypeParameterFixer.kt`. Handles typed query methods where the return type `T` needs runtime type info.

`shouldFix(call)` returns true for:
- `queryList<T>()`, `queryOne<T>()`, `queryOneOrNull<T>()` on `KronosQueryTask`
- Same methods on `SelectClause`, `SelectFrom*` (join results), `SqlHandler`

When matched, injects two arguments at the end of the call:
1. `isKPojo = T::class is subtype of KPojo` (boolean constant)
2. `superTypes = listOf("com.example.Parent", "com.kotlinorm.interfaces.KPojo", ...)` (string list)

This avoids runtime reflection for type checking during result mapping.

## @KronosInit and KClassMapGenerator

`@KronosInit` marks initialization functions (typically `Kronos.init { }`). The compiler plugin generates a `kClassCreator` map inside these functions:

```kotlin
// Generated code (conceptual):
kClassCreator = mapOf(
    User::class to { User() },
    Order::class to { Order() },
    // ... all discovered KPojo classes
)
```

This map enables `fromMapData()` to create instances without reflection.

For cross-module compilation (test module using entities from main module), `initCallSiteLambdas` are post-processed: the call-site lambda is also injected with the kClassCreator map.

## Symbols and Constants

### Symbols.kt (~620 lines)
Resolves all `IrClassSymbol`, `IrSimpleFunctionSymbol`, `IrPropertySymbol` needed by the transformers. Uses `pluginContext.referenceClass()`, `referenceFunctions()`, `referenceProperties()`.

Key symbol groups:
- KPojo interface and its methods
- Field, FunctionField, Criteria, KCascade constructors
- KColumnType, ConditionType, PrimaryKeyType, NoValueStrategyType enums
- Kotlin stdlib: Pair, Map, List, MutableList, String, Boolean, etc.
- KronosCommonStrategy, KTableIndex constructors

### Constants.kt
All `FqName` and `ClassId` constants organized by category:
- Annotations: `Table`, `PrimaryKey`, `Column`, `ColumnType`, `CreateTime`, `UpdateTime`, `LogicDelete`, `Version`, `TableIndex`, `Cascade`, `KronosInit`, `NotColumn`, `Ignore`, `Serialize`, `DateTimeFormat`, `Necessary`, `Default`
- DSL classes: `KTableForSelect`, `KTableForSet`, `KTableForCondition`, `KTableForSort`, `KTableForReference`
- Core types: `Field`, `FunctionField`, `Criteria`, `KCascade`, `KPojo`
- Enums: `KColumnType`, `ConditionType`, `PrimaryKeyType`, `NoValueStrategyType`, `IgnoreAction`

## Debugging IR Output

Enable IR dump via Gradle:
```kotlin
kronos {
    dumpIr = true
    dumpIrPath = "build/tmp/kronosDebug"  // default
    dumpIrMode = "kotlinLike"  // or "common"
    dumpIrFiles = "User.kt,Order.kt"  // comma-separated filter, empty = all
}
```

Or via CLI options to the compiler plugin:
```
-P plugin:kronos-compiler-plugin:dump-ir=true
-P plugin:kronos-compiler-plugin:dump-ir-path=build/tmp/kronosDebug
```

The dump writes transformed IR to files in the specified directory, useful for verifying that transformations produce correct output.
