# IR Cookbook

Concrete patterns for backend IR compiler plugin work.

## Add An IR Generation Extension

Use `IrGenerationExtension` when changing executable output.

Shape:

```kotlin
class MyIrGenerationExtension(
    private val options: MyOptions
) : IrGenerationExtension {
    override fun generate(moduleFragment: IrModuleFragment, pluginContext: IrPluginContext) {
        val symbols = MySymbols(pluginContext)
        val generator = MyIrGenerator(pluginContext, symbols, options)
        moduleFragment.transform(MyModuleTransformer(generator), null)
    }
}
```

Keep symbol lookup and body generation separate. This makes failures easier to locate.

## Organize A Non-Trivial Generator

For complex generation, use this shape:

```text
ir/
  MyIrGenerationExtension.kt
  MySymbols.kt
  MyIrGenerator.kt
  MyBodyGenerator.kt
  MyDeclarationMatcher.kt
```

Responsibilities:

- `MySymbols`: resolve classes, constructors, functions, properties.
- `MyDeclarationMatcher`: identify plugin-generated or annotated declarations.
- `MyIrGenerator`: orchestrate generation.
- `MyBodyGenerator`: create bodies with `DeclarationIrBuilder`.
- Transformer/visitor: traverse module and call generator.

Avoid resolving symbols repeatedly inside hot visitor methods.

## Fill A Body For A FIR-Generated Function

Use FIR for the signature and IR for the body.

Steps:

1. Identify generated function by origin, stable name, owner, annotation, or metadata.
2. Build body with `DeclarationIrBuilder(pluginContext, function.symbol)`.
3. Assign `function.body`.
4. Ensure return type matches exactly.
5. Run IR verification.

Conceptual shape:

```kotlin
fun generateBody(function: IrSimpleFunction) {
    function.body = DeclarationIrBuilder(context, function.symbol).irBlockBody {
        +irReturn(buildReturnExpression(function.returnType))
    }
}
```

Never generate an expression whose type only "looks close". IR verifier and lowerings are stricter than source.

## Rewrite Existing Bodies

Use an IR transformer when source already resolves and existing bodies need instrumentation or replacement.

Patterns:

- `IrElementTransformerVoidWithContext` for context-aware traversal.
- Override `visitCall`, `visitFunctionNew`, `visitClassNew`, or specific expression methods.
- Call `super` unless intentionally replacing a subtree.
- Preserve source offsets where meaningful.

Guard every transform:

- Check fully qualified names.
- Check dispatch/extension receiver types.
- Check origin and symbol owner when matching operators.
- Avoid matching by simple names alone.

## Build Calls Safely

When creating calls:

1. Resolve target symbol once.
2. Set dispatch receiver if target is an instance member.
3. Set extension receiver if target is an extension.
4. Fill value arguments by parameter index.
5. Set type arguments if the function is generic.
6. Use nullable/non-nullable types deliberately.

Common bug: using the wrong argument index after Kotlin compiler API changes. Prefer helper accessors around value parameters.

## Generated Declarations And Origins

When backend needs to find frontend-generated declarations, use stable markers:

- Plugin declaration origin.
- Stable generated names.
- Owner class ID and callable ID.
- Plugin annotations.
- Metadata generated by frontend.

Avoid relying on traversal order.

## Debugging IR

When IR fails:

1. Reduce to one declaration and one generated body.
2. Dump IR before relevant lowerings.
3. Enable IR verification.
4. Inspect receivers, type arguments, and return types.
5. Compare generated IR against equivalent hand-written Kotlin.

If hand-written Kotlin cannot express the construct, be extra cautious. You may be generating invalid IR.
