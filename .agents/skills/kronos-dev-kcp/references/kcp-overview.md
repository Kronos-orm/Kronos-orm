# Kotlin Compiler Plugin Overview

This reference summarizes Kotlin compiler plugin architecture. It is written as operational guidance, not as a link index.

## Stability

Kotlin compiler plugin APIs are not stable. Treat every Kotlin compiler upgrade as an API migration task. Keep version-sensitive code small, well-named, and isolated.

Before choosing a compiler plugin, check whether the task can be solved by:

- KSP source generation.
- A Gradle task.
- A lint/static-analysis tool.
- Ordinary runtime reflection or code generation.

Use a compiler plugin only when the feature must participate in compiler resolution, diagnostics, generated declarations, IR generation, or binary output.

## Pipeline

The simplified pipeline is:

1. Parse source into syntax trees.
2. Build FIR and progressively resolve declarations, imports, annotations, supertypes, types, statuses, bodies, contracts, and constants.
3. Convert FIR to IR.
4. Run IR generation and lowerings.
5. Emit target output.

FIR is incremental by phase. Early FIR is intentionally incomplete and becomes richer over phases such as raw FIR, imports, compiler-required annotations, companion generation, supertype resolution, type resolution, status resolution, body resolve, contracts, implicit body types, and constant evaluation.

Practical phase rule:

- Need only names and declaration shape: early FIR may be enough.
- Need supertypes: use supertype generation or a later phase.
- Need visibility, modality, or modifiers: use status transformation.
- Need expression/body types: use body-resolved FIR or backend IR.
- Need annotation arguments: verify the phase has resolved them.

## Frontend vs Backend

Use FIR/frontend when source must be valid, visible, or diagnosed before backend:

- Generated functions/properties/classes must be visible to user code.
- Invalid source should fail with compiler diagnostics.
- Classes need plugin-added supertypes.
- Declarations need status changes.
- Types need custom attributes.
- Other FIR extensions need a shared session-scoped service.
- IDE highlighting/completion must understand the plugin output.

Use backend/IR when source already resolves and only generated bytecode behavior changes:

- Fill bodies for declarations generated in FIR.
- Rewrite existing bodies.
- Insert calls or expressions.
- Generate target-specific runtime behavior.

Use both when FIR exposes/validates symbols and IR supplies executable implementation. Kotlinx serialization is the useful mental model: frontend exposes generated declarations and diagnostics; backend fills serializer-related bodies and implementation details.

## Project Shape

A compiler plugin project usually separates:

- Compiler plugin implementation.
- User-facing annotations.
- Gradle plugin integration.

The Gradle plugin typically adds the compiler plugin artifact to Kotlin compilation and adds annotation dependencies to user code. Keep annotation artifacts independent from compiler internals so user projects do not depend on compiler classes.

## Registration Shape

Typical implementation pieces:

```text
compiler-plugin/
  plugin/MyCommandLineProcessor.kt
  plugin/MyCompilerPluginRegistrar.kt
  fir/MyFirExtensionRegistrar.kt
  fir/MyCheckers.kt
  fir/MyDeclarationGeneration.kt
  ir/MyIrGenerationExtension.kt
  resources/META-INF/services/...
```

Exact service entries and registration APIs vary by Kotlin version. Check the target Kotlin version before copying code between projects.

## Backend IR Generator Shape

For serious backend work, organize IR generation as a coordinated generator rather than scattered visitor snippets.

A practical generator object holds:

- Plugin context and symbol references.
- Shared type helpers.
- Generated declaration lookup.
- Body-generation helpers.
- Sub-generators for related responsibilities.

This shape is useful when a plugin needs several generated functions/classes, type substitutions, reusable body builders, and coordination between FIR-generated declarations and IR-filled bodies.

## Backend Checks

For backend plugins:

- Use `IrGenerationExtension`.
- Override `generate(moduleFragment, pluginContext)`.
- Reduce test sources to the smallest reproducer.
- Enable IR verification for non-trivial transformations.
- Dump IR around relevant lowering phases.
- Compare generated dumps with a manually understood expected shape.
