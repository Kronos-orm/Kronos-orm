# Testing Kotlin Compiler Plugins

This reference summarizes compiler plugin testing practice from the Kotlin compiler test infrastructure and the Kotlin compiler plugin template.

## Test Layout

Use test-data-driven tests. A typical compiler plugin keeps source tests under:

```text
compiler-plugin/testData
compiler-plugin/testData/box
compiler-plugin/testData/diagnostics
```

Use:

- `box` tests for generated runtime behavior.
- `diagnostics` tests for frontend errors and warnings.
- FIR and FIR+IR dump tests for structural compiler output.

The plugin template can generate or update JUnit test classes from test data. Keep generated runner classes thin; put behavior in test data, directives, configurators, and handlers.

## Write A Box Test

Use a box test when generated code should run and return a result.

Pattern:

```kotlin
// testData/box/myFeature.kt
import my.plugin.annotation.GenerateHello

@GenerateHello
class User

fun box(): String {
    return if (User().hello() == "OK") "OK" else "fail"
}
```

What this proves:

- The plugin was loaded.
- Generated declarations are visible if `hello()` is generated by FIR.
- Backend IR generated a valid executable body.
- Runtime behavior matches expectations.

Add a `.fir.txt` and `.fir.ir.txt` golden when the plugin affects frontend or IR structure. Read the golden diff; do not approve it blindly.

## Write A Diagnostics Test

Use diagnostics tests when invalid source should fail.

Pattern:

```kotlin
// testData/diagnostics/invalidUsage.kt
import my.plugin.annotation.GenerateHello

@GenerateHello
fun notAClass() {}
```

Expected diagnostics should assert the plugin error is emitted at the right source location. Include a valid companion test if the checker has non-trivial filtering logic.

What this proves:

- FIR checker is registered.
- Diagnostic factory and renderer are wired.
- Checker triggers only for intended declarations.

## Test Generated Declarations

For FIR declaration generation, include tests that reference generated declarations from user source:

```kotlin
@GenerateHello
class User

fun box(): String = User().hello()
```

If this does not compile, the problem is usually frontend generation, not IR. Inspect `.fir.txt` first.

Checklist:

- Generated name appears in member callable names.
- Generated declaration has plugin origin.
- Return type and receiver are resolved.
- If the generated declaration has no body, backend IR fills it later.

## Test IR Body Generation

For backend body generation, write a box test and compare IR if the body is complex.

Checklist:

- The generated function is located by stable marker.
- Body is assigned exactly once.
- Return expression type matches function return type.
- Dispatch/extension receivers are correct.
- Generic type arguments are filled.
- IR verifier passes.

## Golden Files

A single source test can have generated dump files such as:

- `.fir.txt`
- `.fir.ir.txt`

The FIR dump shows frontend declarations, calls, symbols, statuses, and type information for the relevant phase. The FIR+IR dump shows backend IR after FIR conversion and plugin participation.

Golden-file rule: update expected dump files only after reading the diff and identifying the compiler phase responsible for each change.

Typical golden-review questions:

- Did a generated declaration appear in the intended FIR phase?
- Did a checker add only diagnostics, or did it accidentally change resolution?
- Did FIR-to-IR preserve generated declarations?
- Did backend IR insert the expected body and no unrelated declarations?
- Did a Kotlin upgrade rewrite dumps without semantic plugin changes?

## Kotlin Test Infrastructure Model

The compiler test infrastructure is pipeline-based.

Core concepts:

- Module: the basic compilation unit. A test can define multiple modules.
- Artifact: result of a pipeline step.
- Facade: transforms one artifact into another, such as frontend output into backend input.
- Handler: validates, dumps, or renders information from an artifact.
- Step: either a facade step or a handler step.
- Directive: test-data comments that configure behavior.
- Test service: strongly typed per-test service shared by facades, handlers, and configurators.
- Environment configurator: modifies compiler configuration based on directives.

Test-data structure:

- `// FILE: name.kt` starts a new file.
- `// MODULE: name` starts a new module.
- Modules can declare regular dependencies, friend modules, and dependsOn relationships.
- Without explicit module/file directives, the test uses a default module and default file.

Directives:

- Simple directive: `// DIRECTIVE`.
- String/value directive: `// DIRECTIVE: arg`.
- Declare directives in directive containers.
- Use configurators to map directives into compiler configuration or analysis flags.

## Test Strategy

For FIR plugins:

- Test invalid source with diagnostics tests.
- Test generated declarations by compiling source that references them.
- Test predicate behavior with annotated, meta-annotated, nested, and unannotated declarations.
- Test phase-sensitive behavior, especially annotation arguments, supertypes, status, and generated declarations.

For IR plugins:

- Test generated runtime behavior with `box` tests.
- Enable IR verification for complex transformations.
- Dump IR around relevant lowering phases.
- Compare generated body shape with expected IR when behavior is subtle.

For registration/options:

- Test that the plugin loads.
- Test CLI/Gradle options are accepted.
- Test frontend and backend parts can both be registered.
- Test option values reach extension constructors or compiler configuration.

## Add A Test Runner

If the project uses Kotlin's compiler test infrastructure, create an abstract generated test class rather than hand-writing many JUnit methods.

High-level shape:

1. Define a test class extending the appropriate compiler-test base.
2. Configure the pipeline with frontend, backend, handlers, and plugin configurators.
3. Point the generated tests at `testData`.
4. Run the generator task when adding new test files.

Keep the runner boring. Most plugin behavior should be expressed by:

- Test data source.
- Directives.
- Expected diagnostics.
- FIR/IR dump files.

## Useful Directives

Use directives for per-test toggles:

```kotlin
// ENABLE_PLUGIN_OPTION: strict=true
// FIR_IDENTICAL
// IGNORE_BACKEND: JS_IR
```

Directive names are project-specific. Define only directives the test configurator actually reads. Remove stale directives; they make test intent ambiguous.

## Debugging Checklist

When a compiler plugin test fails:

1. Determine whether the failure is frontend resolution, diagnostics, FIR-to-IR conversion, backend IR generation, lowering, or runtime.
2. Inspect `.fir.txt` first for frontend issues.
3. Inspect `.fir.ir.txt` or backend IR dump for IR issues.
4. Compare the failing dump section with the expected phase: RAW_FIR, IMPORTS, COMPILER_REQUIRED_ANNOTATIONS, COMPANION_GENERATION, SUPER_TYPES, TYPES, STATUS, BODY_RESOLVE, CONTRACTS, IMPLICIT_TYPES_BODY_RESOLVE, CONSTANT_EVALUATION, or later phases.
5. Reduce the test source to the smallest reproducer.
6. Avoid updating golden files until the compiler phase and source of the change are understood.
