# Task 8: Introduce Mutually Exclusive Page And Cursor Query Stages

Progress: 100%
Status: Done

## Goal

Replace `withTotal().page(...)` and `withCursor().cursor(...)` with typed query stages that expose only valid pagination combinations and return named result types.

## Completed

- Added `OffsetPageQuery`, `TotalPageQuery`, and `CursorPageQuery` on the shared selected-query boundary; the base query is `KSelectable`, and the agreed remaining stage change makes the single-query offset page a `KSelectable` derived source without making total/cursor execution stages selectable.
- Added named `PageResult`, `CursorResult`, and `TotalPageTasks` contracts; page metadata includes `pageIndex` and `pageSize`.
- Ordinary select, derived select, and JOIN create immutable page/cursor snapshots without mutating base or sibling queries.
- Removed legacy pagination types, triples, pre-page `withTotal`, `withCursor`, and cursor `offset` naming from production.
- Total-count planning preserves DISTINCT output cardinality and grouped cardinality for ordinary and JOIN queries.
- Cursor construction now requires explicit field ordering, preserves aliases and hidden-label uniqueness, resolves the default wrapper once, protects Int boundaries, and round-trips supported numeric/UUID/date-time token types.
- JOIN cursor only accepts statically safe inner/cross base-table trees and appends a stable key for every leaf; unsafe shapes fail before SQL execution.
- `OffsetPageQuery` is a finite `KSelectable` derived source; outer select, JOIN, union, page, and cursor layers preserve the inner LIMIT/OFFSET without exposing total or cursor execution wrappers as relations.
- Stable keys propagate conservatively by real output label across physical-column mappings, complete non-null UNIQUE keys, aliases, and multiple derived layers; incomplete or nullable keys remain rejected.
- Generated Selected construction preserves directly selected non-root JOIN fields by captured receiver type across derived and union/selectable layers.
- Derived aliases keep SQL `AS` labels aligned with projection metadata, including multiple renamed layers.

## Remaining

- None.

## Acceptance

- `query.page(1, 20).toList()` returns records without a count query.
- `query.page(1, 20).withTotal().toList()` returns `PageResult<T>` and executes one count plus one record query.
- `query.cursor(pageSize = 20, after = cursor).toList()` returns `CursorResult<T>`.
- Page stages expose no cursor method; cursor stages expose no page/withTotal method; total stages expose no second transition.
- `query.page(1, 20).select { ... }` treats the paged query as a derived source, retains the inner LIMIT/OFFSET, and allows the new outer query to choose its own page or cursor stage.
- `OffsetPageQuery` is a `KSelectable`; `TotalPageQuery` and `CursorPageQuery` are not. Cursor lookahead rows and hidden token columns never leak into a derived source.
- Ordinary select, JOIN select, and derived select use one pagination implementation and result contract.
- Creating or building one page/cursor view does not alter later builds of the base query or another view.
- SQL Server ordering requirements and all five database pagination renderings remain correct.
