# Kronos IntelliJ IDEA plugin

<center>
<img src="/assets/images/features/img-2.png" width="300"/>
</center>

--------

Kronos DSL code depends heavily on compile-time Kotlin information. `select { ... }` generates projection fields for result rows, a query result can become the input type of another `.select { ... }`, and `orderBy { ... }` receives the fields that are sortable in the current query.

The IntelliJ IDEA plugin feeds that generated information into IDEA K2 analysis, so completion, type presentation, and red-line feedback in the editor follow the same rules as compilation.

## What the plugin supports

The current plugin focuses on these editing scenarios:

- Resolving generated `KPojo` members such as `toDataMap`, `fromMapData`, field metadata, and dynamic accessors.
- Resolving result-row fields generated by `select { ... }`.
- Resolving the input type used when a query result continues with `.select { ... }`.
- Offering original fields and aliases generated by the current query in `orderBy { ... }`.
- Reporting the currently available fields in `where`, `groupBy`, and `having`.
- Showing DSL diagnostics for missing aliases, duplicate projection names, scalar subquery shape errors, predicate subquery column-count errors, and insert-select value count or type mismatches.

## Generated projection fields

The query below involves three shapes: the user-defined `User`, the generated result row `UserNameLengthRow`, and the generated ordering context `UserNameLengthOrderContext`.

```kotlin
data class User(
    var id: Int? = null,
    var name: String? = null,
    var status: Int? = null,
) : KPojo

data class UserNameLengthRow(
    var id: Int? = null,
    var nameLength: Int? = null,
) : KPojo

data class UserNameLengthOrderContext(
    var id: Int? = null,
    var name: String? = null,
    var status: Int? = null,
    var nameLength: Int? = null,
) : KPojo
```

```kotlin
val query = User()
    .select {
        // it: User
        // Generates UserNameLengthRow(id, nameLength)
        [it.id, f.length(it.name).alias("nameLength")]
    }
    .where {
        // it: User
        it.status == 1
    }
    .orderBy {
        // it: UserNameLengthOrderContext
        it.nameLength.desc()
    }

val rows = query.queryList()
// rows: List<UserNameLengthRow>

rows.first().id
rows.first().nameLength
```

In code like this, IDEA can complete and analyze `id` and `nameLength`. The `queryList()` result follows the current result-row shape, including projection fields generated during compilation.

The `where` in the same query reads input fields. An alias selected in that query can be filtered after the query result is used as input. IDEA reports `nameLength` in the following code:

```kotlin
User()
    .select {
        // it: User
        [it.id, f.length(it.name).alias("nameLength")]
    }
    .where {
        // it: User
        it.nameLength > 8
    }
    .queryList()
```

When the query result is used as input, `it` is the generated result row:

```kotlin
val q = User()
    .select {
        // it: User
        [it.id, f.length(it.name).alias("nameLength")]
    }

q.select { [it.id, it.nameLength] }
    .where {
        // it: UserNameLengthRow
        it.nameLength > 8
    }
    .queryList()
```

## Window result filtering

Window function aliases can be used by `orderBy` in the same query. When you need to filter by a window result, use the current query result as another query input.

```kotlin
val ranked = Order()
    .select {
        // it: Order
        [
            it.id,
            it.userId,
            f.rowNumber()
                .over {
                    // Still references fields from the outer it: Order
                    partitionBy(it.userId)
                    orderBy(it.createTime.desc())
                }
                .alias("rn")
        ]
    }

ranked
    .select {
        // it: RankedOrderRow
        [it.id, it.userId]
    }
    .where {
        // it: RankedOrderRow
        it.rn == 1
    }
    .queryList()
```

If you write the version below in the same query, IDEA marks `rn` as invalid:

```kotlin
Order()
    .select {
        // it: Order
        [
            it.id,
            f.rowNumber()
                .over { orderBy(it.createTime.desc()) }
                .alias("rn")
        ]
    }
    .where { it.rn == 1 }
```

## Alias diagnostics

Direct fields can inherit their field names. Functions, aggregates, window functions, scalar subqueries, and computed expressions must declare `.alias("name")` when they are used as select items.

```kotlin
User()
    .select { [it.id, f.length(it.name)] }
```

IDEA reports that the non-field select item needs `.alias("name")`.

```kotlin
User()
    .select { [it.id, f.length(it.name).alias("id")] }
```

This reports a duplicate selected field name. The next example also fails because the new alias conflicts with an original field, making the ordering context ambiguous:

```kotlin
User()
    .select { [it.id, f.length(it.name).alias("name")] }
```

## Scalar subquery diagnostics

A scalar subquery must select exactly one column and usually needs explicit `.limit(1)`. It also needs an alias when used as a select item.

```kotlin
User()
    .select { u ->
        [
            u.id,
            Order()
                .select { it.amount }
                .where { it.userId == u.id }
                .orderBy { it.createTime.desc() }
                .limit(1)
                .alias("lastAmount")
        ]
    }
```

These examples are reported in the editor:

```kotlin
// Missing alias
User().select { u ->
    [u.id, Order().select { it.amount }.where { it.userId == u.id }.limit(1)]
}

// Missing limit(1)
User().select { u ->
    [
        u.id,
        Order()
            .select { it.amount }
            .where { it.userId == u.id }
            .alias("lastAmount")
    ]
}

// More than one selected column
User().select { u ->
    [
        u.id,
        Order()
            .select { [it.amount, it.status] }
            .where { it.userId == u.id }
            .limit(1)
            .alias("lastOrder")
    ]
}
```

## Predicate subquery diagnostics

The plugin also reports shape errors for `IN`, `EXISTS`, `ANY`, `SOME`, `ALL`, and row-value tuple predicates.

```kotlin
User()
    .select()
    .where { user ->
        user.id in Order().select { it.userId } &&
            exists(Order().select().where { it.userId == user.id })
    }
```

These forms are invalid:

```kotlin
// Single-value IN requires a single-column subquery
User().select().where {
    it.id in Order().select { [it.userId, it.status] }
}

// Row-value tuple arity must match the subquery arity
User().select().where {
    [it.id, it.status] in Order().select { it.userId }
}

// Single-element tuples are rejected; use it.id in query
User().select().where {
    [it.id] in Order().select { it.userId }
}

// any/some/all also require a single-column right-hand query
Order().select().where {
    it.status > any(Order().select { [it.status, it.amount] })
}
```

## Insert-select feedback

The lambda receiver of `insert<Target> { ... }` comes from the result row generated by the source query. The IDEA plugin lets you access source projection fields there and reports value count or type mismatches.

```kotlin
data class UserOrderSummary(
    var userId: Int? = null,
    var orderCount: Long? = null,
) : KPojo

Order()
    .select { [it.userId, f.count(it.id).alias("orderCount")] }
    .groupBy { it.userId }
    .insert<UserOrderSummary> { [it.userId, it.orderCount] }
    .execute()
```

If you provide too few fields, or if the order maps fields to the wrong types, the editor points at the values list:

```kotlin
val orderCounts = Order()
    .select { [it.userId, f.count(it.id).alias("orderCount")] }
    .groupBy { it.userId }

orderCounts.insert<UserOrderSummary> {
    [it.userId]
}

orderCounts.insert<UserOrderSummary> {
    [it.orderCount, it.userId]
}
```
